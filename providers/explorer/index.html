Interface

MemoryProvider — the canonical ABC that all memory adapters must implement.

store() query() update() delete() get() health() capabilities()

Conformance Suite

T01–T15: health_ok, health_format, store_basic, store_with_metadata, query_semantic, query_empty, query_with_filters, update_content, update_preserves_id, delete_basic, delete_nonexistent, capabilities_declared, error_on_invalid, latency_budget, capabilities_type.

Lifecycle Status

Gold Standard — L3 (Production). The ground truth reference. Every other memory provider must pass the same T01–T15 conformance suite. Changes to MemoryProvider ABC or InMemory implementation require an RFC.

When to use

Development & Testing: Use InMemory as your go-to provider during development and unit tests. No dependencies, no configuration, zero latency. It validates behaviour before you plug in a persistent backend. Also acts as the contract enforcer — if it works on InMemory but fails on Mem0, the adapter is the problem, not your code.

Conformance Suite

All 15 tests (T01–T15) pass. Identical test class to InMemory's, run against the Mem0 adapter + mock server. No regressions allowed — if InMemory passes but Mem0 doesn't, the adapter is blocked from production.

Lifecycle Status

Active — L3 (Production). Fully conformant memory provider for production workloads. Uses the external Mem0 API which provides persistent, queryable memory storage. Top-tier maturity alongside InMemory.

When to use

Persistent production memory: When you need memory that survives restarts, survives across sessions, and supports semantic querying with metadata filtering. Mem0 is the primary persistent MemoryProvider for production AI agents. Choose it over InMemory when data durability matters. Requires a running Mem0 API endpoint.

Interface

KnowledgeProvider — the ABC for read-access to structured knowledge stores. Different from MemoryProvider: no store/update/delete, focused on search, list, and retrieval.

search() get() list() health() capabilities()

Conformance Suite

TK01–TK43: 43 tests covering health, listing, searching (by tags, content, path), filtering, frontmatter metadata, cross-file concept search, incremental reads, pagination, error handling, edge cases (empty vault, special characters), and latency budgets. The largest conformance suite of any provider.

Lifecycle Status

Frozen v1. Reference implementation is locked. The interface, conformance tests, and behaviour are stable. No changes permitted without an RFC. The vault-access research is complete and frozen.

When to use

Reading structured knowledge from an Obsidian vault: Use Obsidian when you need your AI agent to search, list, and retrieve markdown notes from a local Obsidian vault. It supports frontmatter metadata, tag-based filtering, cross-file content search, and paginated reads. Does NOT support writing — this is a read-only knowledge provider by design. Ideal for agents that need access to a curated knowledge base.

Unlike Obsidian (general vault knowledge), Skills is specifically for agent-internal operating procedures. It's a lightweight, file-system-backed knowledge provider.

Archive providers/archive/adapter.py
Frozen v1 KnowledgeProvider

Conformance ● 21/21 (TA01-TA21)
Lifecycle Frozen v1
Backend JSONL archive (local)
search get list store
Frozen
v1

Interface

KnowledgeProvider — the only KnowledgeProvider that also supports store(). Provides read/write access to an archived memory store for historical records (JSONL format). Designed as a write-once, read-many cold storage layer.

search() get() list() store() health() capabilities()

Conformance Suite

TA01–TA21: 21 tests covering health, storing memories, retrieval, date-based filtering, importance filtering, tag filtering, partial content matching, pagination, error handling (empty store, malformed entries, nonexistent lookups), and read-after-write consistency. See ADR-0012.

Lifecycle Status

Frozen v1. Archive was the last provider frozen. It fills the gap for historical/cold memory — records that should persist but aren't needed for daily semantic queries. The store() capability makes it unique among KnowledgeProviders.

When to use

Cold storage for historical memories: Use Archive when you need to persist memory records that are no longer in active use but must be retained for audit, compliance, or historical reference. It's a write-once, read-rarely layer — the attic of the memory platform. Unlike InMemory/Mem0 (hot storage for active agent memory), Archive is for data that should outlive sessions but only needs occasional retrieval. Supports filtering by date, importance, and tags.
