🏗️ Platform Overview

Why Memory ≠ Knowledge — the architecture, philosophy, and problems the Memory Platform solves.

🤔 Why Memory ≠ Knowledge

This is the core insight of the Memory Platform. Most systems treat memory and knowledge as the same thing — they're not.

Dimension 🧠 Memory 📚 Knowledge
What it is Personal, episodic — what this agent experienced Universal, semantic — facts about the world
Source Conversations, actions, decisions, user feedback Documentation, code, APIs, training data
Scope Per-user, per-session, per-agent Shared across all agents and users
Volatility 🟡 Medium — relevance decays, needs consolidation 🟢 Low — stable, versioned, curated
Query pattern Similarity search + recency + importance Keyword / semantic search on curated corpus
Example "Last week the user asked for dark mode" "The API accepts a `theme` parameter"

⚠️ The Common Mistake

Storing ephemeral conversation memories in the same vector store as curated documentation leads to context pollution — the agent retrieves outdated personal preferences when it needs factual knowledge, or misses personal context because it's buried in generic docs. The Memory Platform separates these concerns explicitly.

🔁 How the Pipeline Works

  1. Action Router — receives the request and routes it to the correct handler (memory.store, context.build, etc.)
  2. Memory Store — personal, episodic memories from previous interactions
  3. Knowledge Store — curated facts, docs, skills, and archival knowledge
  4. Policy Engine — filters low-confidence memories before storage (importance ≥ threshold)
  5. Context Builder — merges results from all providers, ranked by relevance score
  6. AI Brain — consumes the enriched context to produce the final answer
  7. Explainability Trace — every decision is logged with a trace_id

🎯 Problems It Solves

🧩 Problem 1: Context Pollution

When all data is stored together, personal memories drown out factual knowledge and vice versa. The Memory Platform's two-tier architecture (Memory vs Knowledge) keeps them cleanly separated until the Context Builder intelligently merges them.

📉 Problem 2: Relevance Decay

Old memories shouldn't have the same weight as new ones. The platform uses recency scoring × importance thresholds to ensure yesterday's unimportant chat doesn't override today's critical context.

🔌 Problem 3: Provider Lock-In

Switching from InMemory to Mem0 to Honcho shouldn't require rewriting your application. The Provider ABC contract (MemoryProvider, KnowledgeProvider) means any provider plugs in with zero application changes.

🔍 Problem 4: Opaque Decisions

Why did the agent answer that way? Every response includes an explainability trace with trace_id showing exactly which memories and knowledge were used, which policy rules fired, and what was filtered out.

✅ Problem 5: No Quality Gate

Without a policy engine, low-confidence or irrelevant memories get stored and later retrieved — polluting future answers. The Policy Engine evaluates importance BEFORE storage, discarding memories below threshold.

📐 Design Principle

Never optimize before correctness. The Golden Dataset (200 test cases) is the final judge — any change that causes a >5% regression in benchmark scores is automatically rejected.
